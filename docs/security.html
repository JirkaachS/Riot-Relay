Security model

Layered local protection.

Riot Relay reduces account mix-ups and protects secrets at rest. It is not a substitute for device security, Riot multi-factor authentication, or careful release verification.

Encrypted vault

  • Passwords are sealed with AES-256-GCM, which provides confidentiality and tamper detection.
  • The key is derived from the master password with scrypt and a per-vault random salt.
  • Only ciphertext, salt, nonce, and authentication tag are intentionally persisted for account secrets.
  • OS key storage begins disabled and can be enabled only from an unlocked vault.
  • Stored keys are bound to the corresponding vault and unlock mode; Windows protection is device/user-context dependent.

Choose a unique master password. There is no hosted recovery path. Malware or a person controlling your unlocked Windows session may still access data while the vault or Riot session is active.

Safe identity binding

Riot display names can change and labels are user-defined, so they are not sufficient security identifiers. Riot Relay treats the Riot PUUID as authoritative. Sync and capture flows compare the currently authenticated Riot identity before attaching sessions, ranks, inventory, portraits, or account metadata to a roster entry.

Identity mismatch should cause the operation or returned data to be rejected instead of silently crossing account boundaries. For League rank providers, exact identity checks are applied and verified local client data is used as a fallback where available.

Never edit local vault/session files to force a binding. Sign into the intended account through Riot Client, sync it, verify the shown Riot ID, and only then save or use its session.

Build and download trust

Official project downloads are attached to GitHub Releases. Tagged release automation runs the test suite and invokes the repository’s x64 NSIS and MSI release scripts. The EXE is recommended for automatic updates; MSI is for managed or manual updates.

Unsigned software: the project does not claim code signing. Windows SmartScreen may identify a build as unrecognized. Confirm the download came from JirkaachS/Riot-Relay, review the tag/source and published checksums when available, and do not bypass a warning for a copy obtained elsewhere.

Open source and automated workflows improve inspectability but do not prove that every downloaded binary is harmless. Build from reviewed source when your threat model requires it.

Security boundaries

  • Riot Relay needs access to active Riot sessions and locally decrypted credentials to perform requested actions.
  • Encryption at rest does not protect an unlocked vault from malware, screen capture, injected code, or an attacker controlling the same Windows account.
  • Optional Windows-protected unlock trades repeated password entry for reliance on Windows account/device protection.
  • Third-party and undocumented APIs may change, return incorrect data, log network metadata, or become unavailable.
  • Presence controls alter published chat presence; they do not hide gameplay, account activity, or network traffic from Riot.
  • Riot can change authentication or client behavior. Never disable Riot security controls to make switching work.

Report a vulnerability

Use the repository’s private vulnerability reporting feature if it is available. Otherwise, open a minimal GitHub issue asking maintainers for a private contact channel. Do not publish an exploitable proof, tokens, passwords, vault files, session files, private logs, or real Riot account identifiers.

Include the affected version, operating system, impact, reproducible steps using synthetic data, and a proposed mitigation if known. For ordinary usage failures, use troubleshooting.
